多级菜单的后端设计#

多级菜单在日常前端页面中有较多应用，如商城顶部分类导航栏以及省市区三级菜单联动等。此文章将从后端角度来考虑商城顶部分类的多级联动菜单的设计。

后端层面的设计需要考虑多级菜单在后端代码中的表现形式以及在数据库中的存储格式，以及如何进行从数据库到后端的数据转换，接下来将阐释我的设计思路。

方式一#

后端数据结构#

考虑到多级菜单是单向的从上到下的一个结构，比较类似N叉树，所以采用N叉树来表示菜单信息。表示出来如下图：

  1. 第一行表示分类id
  2. 第二行表示分类名称
  3. 第三行表示父分类id

代码

@Data
public class TreeNode {
    private String data; // 此节点菜单名称 
    private List<TreeNode> children;
}

数据库设计#

数据库方面我们选用mysql来存储表单信息。由于mysql是一种关系型数据库，存储的记录是基于行的，所以我们需要将N叉树展开存储，可以使用邻接表来进行存储。

邻接表可以表示当前节点与与哪些节点相连，我们可以选择存储当前节点的子节点列表或者存储当前节点的父节点。此处我们选择存储当前节点的父节点，因为子节点不确定有多少个，而有向的树结构可以确定一个节点一定只有一个或多个子节点。

create table test_category
(
    ID        INT(3) AUTO_INCREMENT PRIMARY KEY,
    NAME      VARCHAR(20) NOT NULL COMMENT '类别名',
    PARENT_ID INT(3) COMMENT '父类别ID，当是最高层级是为null'
);

优缺点#

优点#

结构较为简单，且增加、修改类目时较为简单。比如要在服装类目下新增一个子类：鞋子，那么只需要新建一条记录，将PARENT_I赋值为服装的ID即可。修改时修改对应类目的NAME以及PARENT_ID即可。

缺点#

查询某个节点下的子节点时会有些力不从心。如果我们不是每次查询所有，而是只单独查询服装类目以及其下的子类目，那么我们需要递归查询服装类目的子类目以及其子类目的子类目…,当层次较深以及结构较多时，比如公司的员工信息管理，员工的职级层级可能会比较深，且员工数量较多，这种设计方法的效率就亟待优化了。

方式二#

结构改动#

方式一可以较为简便地实现一个多级菜单后端结构，但是对于层级结构复杂以及数据库内容增多时，或者要根据某个层级查询下层以及孙子层等方面的需求显得不是那么适用：

  1. 条目增多时，查询所有记录并转换这个方法会占用较多内存以及耗时；
  2. 如果要查询某个节点的子节点以及孙子节点，需要递归进行查询。

要解决上述问题，可以给每个节点加两个“锚点”。

“锚点"类似B树，每个数节点会保存子节点的范围，但我们保存的是子节点的锚点范围。

下图即为改进后的数据结构：

  1. 提供一个哑结点，为最上层分类的上层，将其整合为一个树，而不是之前的多个树
  2. 增加left以及right分别代表左锚点与右锚点

如何保证父节点的锚点可以包含所有子节点以及根节点的锚点范围呢？我们需要使用前序遍历来对锚点进行赋值，步骤如下：

  1. 声明锚点索引为1
  2. 在入树节点时为左锚点赋值，在出树节点时为右锚点赋值，每次为锚点赋值后，锚点索引加一。

为什么前序遍历可以保证父节点包含所有子节点的锚点范围呢？ 下图可以解释这个过程：

以下即为两个"锚点"赋值的流程图，从哑结点开始，每次进入节点时为left赋值，出节点时为right赋值，每个节点中的数字代表他们的前序遍历顺序。

数据库改动#
create table test_category
(
    `ID`        INT(3) AUTO_INCREMENT PRIMARY KEY,
    `NAME`      VARCHAR(20) NOT NULL COMMENT '类别名',
    `PARENT_ID` INT(3) COMMENT '父类别ID，当是最高层级是为null',
    `LEFT`      INT(3)      NOT NULL COMMENT '左边界',
    `RIGHT`     INT(3)      NOT NULL COMMENT '右边界'
);

如何简化查询#

采用前序遍历的方式对左右锚点进行赋值，根据前序遍历的特点，当前节点的子节点以及孙子节点的left一定大于该节点的left，其子节点以及孙子节点的right一定小于该节点的right。

前序遍历顺序为 左 -> 根 -> 右 我们在进入节点时对left赋值，出此节点时对right赋值，可以保证以该节点为根节点的树中，此节点一定是第一个进入(即在树中根节点的left最小)并且最后一个出(此子树中的right最大)。

所以我们可以采用以下方式来查询某个分类下的所有分类：

-- 设我们要查询id为3的分类下面的所有子分类以及孙子分类
select toFind.*
from test_category downline
join test_category toFind
on downline.`LEFT` > toFind.`LEFT`
and downline.`RIGHT` < toFind.`RIGHT`
where toFind.ID = 3

也可以用来查询某个分类的所有父分类

-- 设我们要查询id为3的分类的所有父分类
select toFind.*
from test_category downline
join test_category toFind
on downline.`LEFT` < toFind.`LEFT`
and downline.`RIGHT` > toFind.`RIGHT`
where toFind.ID = 3

优缺点#

优点#
  1. 便于统计某一个节点的所有子节点以及父节点
  2. 方便统计某个节点下有多少个子节点：(right - left - 1) / 2，因为每一个子节点都有两个锚点，所以可以如此计算出子节点的数量
  3. 可以方便地判断当前类目下是否有子类目：如果right - left == 1则代表是最底层的类目
缺点#

新增删除节点时需要对此节点的遍历序后面的所有节点都需要对左右锚点重新赋值

-- 设需要新增的子类别的left = 3,right = 4
update test_category set `LEFT` = `LEFT` + 2 WHERE `LEFT` > 3;
update test_category set `RIGHT` = `RIGHT` + 2 WHERE `RIGHT` > 4;

因为新增的节点为占据两个位置：left right 所以后面的left和right都需要加2

删除节点时将大于left以及right的分别减去2即可

-- 设需要删除的子类别的left = 3,right = 4
update test_category set `LEFT` = `LEFT` - 2 WHERE `LEFT` > 3;
update test_category set `RIGHT` = `RIGHT` - 2 WHERE `RIGHT` > 4;

总结#

对于层级较少的情况，可以考虑方式一，结构简单易于维护，对于层级较多以及节点数量较多的情况下推荐使用方式二，具体还是要根据业务情况来判断。
