## 什么是分布式锁
在分布式 / 微服务集群环境下，为解决多服务、多进程、多机器间共享资源并发竞争问题，实现同一时间仅有一个请求 / 线程能访问或操作临界资源，从而防止并发冲突、数据脏写、重复操作的跨节点互斥锁机制。

### 通用实现方式
核心原理：依赖一个全局唯一的单点组件统一管控锁资源，所有服务节点统一向该组件申请加锁、释放锁，把分布式环境的并发竞争，转化为单点内的互斥竞争。

生产上通用的两种实现方案：
1. mysql悲观锁
利用 SELECT ... FOR UPDATE 排他行锁实现互斥，依靠事务提交 / 回滚自动释放锁；仅适用于数据已存在的场景，存在间隙锁、并发阻塞、性能偏弱等问题。
2. redis分布式锁
基于 SET key value NX EX 原子命令实现，利用 NX 互斥特性保证同一 key 只能被一个客户端抢占，结合过期时间防止服务宕机导致死锁，高并发场景最常用。

### 何时使用mysql锁
MySQL 悲观锁依赖数据库事务特性，事务提交或回滚后会自动释放锁，实现简单、开发成本低，适合快速落地。但该方案存在明显场景限制，并非所有业务都适合使用。
当锁定条件为**唯一索引**时，数据库可以精准定位数据并添加**行锁**，加锁效率高；**但该方案仅适用于数据记录大概率已存在的场景**。在 InnoDB 默认的「可重复读」隔离级别下，若查询数据不存在，**行锁会退化为间隙锁**，用于规避幻读问题。原本精准的单行互斥，会变成范围锁定，导致大量并发请求被阻塞串行执行，严重压低接口并发性能。
因此，MySQL 悲观锁适用场景：额度扣减、订单状态流转、资产变更等「数据一定存在」的业务。其余高并发、数据新增、记录不一定存在的场景，建议优先使用 Redis 分布式锁。
